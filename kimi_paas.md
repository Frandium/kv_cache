# 月之暗面（Moonshot AI）和清华大学最新研究：推理吞吐量暴涨 54%

紧跟 Kimi K2.6，推一篇有点脑洞的论文，来自月之暗面（Moonshot AI）和清华大学的最新联合研究。

## 一句话说清楚：这论文在搞什么？

把 Prefill（预填充）变成一种跨数据中心的云服务。

换句话说：过去大模型推理的 Prefill 和 Decode 两个阶段必须待在同一个机房里，因为中间要传输的 KVCache 太大，跨机房根本搬不动。论文认为，新一代混合注意力模型的 KVCache 缩小了十几倍甚至几十倍，于是可以把 Prefill 拆出去放到另一处高算力集群上跑，再用普通以太网把 KVCache 传回来做 Decode。

这个架构叫做 Prefill-as-a-Service（PrfaaS）。实测吞吐量：

- 相比同构 PD 部署提升 54%
- 相比朴素异构方案提升 32%

## 为什么要搞跨数据中心？

### 背景：PD 分离是标准范式

PD 分离（Prefill-Decode Disaggregation）已经是大规模 LLM 推理的标准范式。

- Moonshot AI 自家的 Mooncake 系统是这一方向的先行者
- 后续也与 vLLM、SGLang、Dynamo 等做了深度合作，把 KVCache 当成“VIP”来管理

PD 分离的直觉很简单：

- Prefill 是计算密集型
- Decode 是内存带宽密集型

理论上应该用算力强的芯片专门跑 Prefill、用带宽大的芯片专门跑 Decode（异构推理）。

### 痛点：KVCache 传输把 Prefill/Decode 绑死

现实问题出在 KVCache 传输上。

在传统 Dense Attention 模型里，一个 32K token 的请求，单个 MiniMax-M2.5 实例产生的 KVCache 传输速率高达约 60 Gbps；跨数据中心以太网带宽往往扛不住，因此 Prefill 与 Decode 只能共享同一个高带宽 RDMA 网络，被绑在同一个机房里。

（图：MiniMax-M2.5 在不同输入长度下的 KV 吞吐量）

这就导致一个尴尬局面：想搞异构推理可以，但不同类型芯片必须塞进同一个 RDMA 集群，运维僵化，Prefill/Decode 的硬件比例也很难灵活调整。

## 混合注意力模型改变了游戏规则

论文指出关键转折点：新一代混合注意力架构正在从根本上改变 KVCache 的大小。

什么是混合注意力？简单说：只保留少量全注意力层（Full Attention），大部分层用线性注意力（Linear Attention）或滑动窗口注意力（SWA）替代；这些层产生的状态大小固定，不会随输入长度线性增长。

论文里列出一组混合注意力模型在 32K 输入下的 KV 吞吐量对比：

| 模型 | 架构比例 | KV 吞吐量 @ 32K |
| --- | --- | ---: |
| MiniMax-M2.5（Dense） | 全 GQA | ~60 Gbps |
| Qwen3-235B（Dense） | 全 MLA | ~33 Gbps |
| Qwen3.5-397B | 3:1 线性:全注意力 | ~8 Gbps |
| MiMo-V2-Flash | 5:1 SWA:全注意力 | ~4.7 Gbps |
| Ring-2.5-1T | 7:1 线性:全注意力 | 更低 |

从 60 Gbps 直接降到 4.7 Gbps，约 13 倍；Ring-2.5-1T 则通过 MLA + 7:1 混合比例实现了约 36 倍的 KV 内存节省。

这个数量级的变化意味着：KVCache 终于可能通过普通以太网跨数据中心传输。

但论文也强调：模型让跨数据中心传输变得“可能”，要让它“实用”，还需要系统层面的设计。真实负载是突发的，请求长度分布不均，前缀缓存命中不平衡，跨集群带宽还会波动；如果把所有 Prefill 都扔到远端集群，仍然可能拥塞排队、利用率低下。

## PrfaaS 的核心设计

PrfaaS 的核心思想是“选择性卸载”：只把值得的请求送到远端。

### 三个子系统

1. 计算子系统
   - PrfaaS 集群：高算力硬件（如 H200），专门处理长上下文 Prefill
   - 本地 PD 集群：常规硬件（如 H20），负责短请求的 Prefill + 所有请求的 Decode
2. 网络子系统
   - 集群内部：RDMA 高带宽互联
   - 集群之间：普通以太网（VPC 对等连接或专线）
3. 存储子系统：混合前缀缓存池

### 混合前缀缓存池：两类 KVCache 的分组管理

混合注意力模型里有两种不同的 KVCache：

- 线性注意力层的递归状态：大小固定，只能精确匹配复用
- 全注意力层的 KVCache：随长度线性增长，支持前缀部分匹配

（图：混合前缀缓存池架构）

PrfaaS 把这两类 KVCache 分组管理，但共享底层内存池。缓存块分为两类：

- 前缀缓存块：可跨请求复用
- 传输缓存块：传完即丢

全局 KVCache 管理器维护所有集群的缓存元数据，调度器据此决定请求路由。

## 关键调度策略：双时间尺度调度

这是论文最硬核的部分。PrfaaS 的调度器分两个层面运作。

### 短期调度：带宽感知 + 缓存感知路由

设一个长度阈值 t：请求的增量 Prefill 长度（去掉缓存命中的前缀后）超过 t 的，发到 PrfaaS 集群；不超过的，留在本地 PD 集群处理。

直觉是：短请求的 Prefill 往往更偏内存瓶颈，送到高算力集群反而浪费；同时短请求的 KV 吞吐量相对更高，更容易吃满跨集群带宽。

调度器还会实时监控 PrfaaS 集群的出口链路利用率和队列深度：

- 带宽紧张：各集群前缀缓存独立评估，尽量减少跨集群传输
- 带宽充裕：全局最优缓存匹配，甚至允许跨集群缓存迁移

### 长期调度：流量驱动的资源再分配

本地 PD 集群内的 Prefill/Decode 实例比例可以动态调整；当流量模式变化时，调度器会重新计算最优的 Np/Nd 比例与路由阈值 t。

## 实验结果：54% 吞吐量提升

论文用内部一个 1T 参数的混合架构模型（基于 Kimi Linear 架构，3:1 KDA:MLA 层比例）做了案例研究。

### 硬件配置

- PrfaaS 集群：32×H200（高算力，专跑长上下文 Prefill）
- 本地 PD 集群：64×H20（常规 PD 模式，800 Gbps RDMA）
- 跨集群带宽：约 100 Gbps VPC 网络
- 对比基线：96×H20 的同构 PD 集群

### 工作负载

- 输入长度：截断对数正态分布，均值约 27K tokens，范围 128～128K
- 输出长度：固定 1024 tokens
- SLO：40 tokens/s

（图：最优参数搜索过程 / 路由阈值搜索）

### 最优配置

- 路由阈值：t = 19.4K tokens
- 本地 PD 集群：3 个 Prefill 实例 + 5 个 Decode 实例
- 约 50% 请求（长请求）被卸载到 PrfaaS 集群

### 核心结果

| 指标 | PrfaaS-PD | 同构 PD | 朴素异构 PD |
| --- | ---: | ---: | ---: |
| 吞吐量提升 | 基准 | 低 54% | 低 32% |
| P90 TTFT | 基准 | 高 64% | - |
| 跨集群带宽消耗 | 13 Gbps | 不适用 | 更高 |

PrfaaS 集群的平均出口带宽仅 13 Gbps，只占 100 Gbps 以太网链路的 13%，说明混合注意力模型的 KVCache 跨数据中心传输不仅可行，而且还有巨大余量。

朴素异构方案（不做选择性卸载，把所有 Prefill 都扔到 H200）只提升 16% 吞吐量，显著弱于 PrfaaS-PD 的 54%，凸显调度策略的重要性。

## 对未来的影响

这篇论文背后的信号非常明确：

1. 模型架构正在重塑推理系统设计
   - Kimi Linear、Qwen3.5、MiMo-V2-Flash、Ring-2.5-1T 等新一代模型走混合注意力路线
   - KVCache 急剧缩小，让跨数据中心推理从“不可能”变成“值得优化”
2. 硬件专用化趋势加速
   - Rubin CPX 专攻 Prefill 吞吐，LPU 专攻 Decode 带宽，HC1 主打超高内存带宽
   - PrfaaS 让异构硬件可以独立部署与扩缩容，不必硬塞进同一个 RDMA 集群
3. 大规模部署的成本优化空间巨大
   - 即使万卡级部署，PrfaaS 的跨数据中心带宽需求也在 Tbps 量级，现代数据中心可承载
   - 企业可在算力便宜的地方部署 Prefill 集群，在离用户近的地方部署 Decode 集群

## 总结

核心洞察：下一代模型的 KVCache 足够小，小到可以跨数据中心传输。

但仅“够小”不够，还需要选择性卸载、带宽感知调度、缓存感知路由等系统设计配合，模型架构与系统设计双管齐下，才能让跨数据中心异构推理真正落地。

作为 Mooncake 的延续之作，这篇论文体现了 Moonshot AI 在推理系统上的积累；文中也明确提到与 vLLM、SGLang 的合作，相关想法很可能逐步落地到开源推理框架中。
